# Page Content Assembly

Three operations write or modify wiki pages. Each has distinct rules for
frontmatter and body.

| Operation | Source | Body | Frontmatter |
|-----------|--------|------|-------------|
| Direct ingest | File on disk | Preserved as-is | Preserved or generated |
| Enrichment | LLM-produced JSON | Never touched | Fields merged in |
| Query result | LLM-produced JSON | Written by LLM | Generated by wiki |

The wiki owns frontmatter. The LLM never writes YAML frontmatter directly.

---

## Anatomy of a Wiki Page

```
concepts/mixture-of-experts.md
─────────────────────────────────────────────────
---                                 ← frontmatter open
title: "Mixture of Experts"
summary: "Sparse routing of tokens…"
tldr: "Sparse routing of tokens…"
read_when:
  - "Reasoning about MoE architecture"
status: active
last_updated: "2026-04-13"
type: concept
tags: [transformers, scaling]
sources: []
confidence: high
contradictions: []
claims: []
---                                 ← frontmatter close
                                    ← blank line (required)
## Overview                         ← body starts here

MoE routes tokens to sparse expert subnetworks…
```

The file is always: `frontmatter block` + `blank line` + `body`.

## Direct Ingest — Frontmatter Generation

When `wiki ingest <path>` processes a file:

**File has valid frontmatter with `title`** → frontmatter preserved as-is.
`last_updated` is set to today. All other fields kept unchanged.

**File has no frontmatter** → minimal frontmatter generated:

| Field | Value |
|-------|-------|
| `title` | First H1 heading in body, or filename stem if no H1 |
| `summary` | Empty — to be enriched later |
| `status` | `active` |
| `last_updated` | Today's ISO 8601 date |
| `type` | Inferred from category prefix (`concept`, `source-summary`, etc.) or `concept` as default |
| `tags` | `[]` |
| `sources` | `[]` |
| `contradictions` | `[]` |
| `claims` | `[]` |

The body is written exactly as found in the source file — no reformatting,
no injection.

## Enrichment — Frontmatter Merge Rules

When `wiki ingest --analysis enrichment.json` processes an `Enrichment` entry,
the existing page's frontmatter is read, fields are merged, and written back.
**The body is never read or written.**

Merge rules per field:

| Field | Rule | Rationale |
|-------|------|-----------|
| `tags` | UNION — accumulate, never shrink | Tags from prior ingests must not be silently removed |
| `read_when` | UNION — accumulate, never shrink | Retrieval hints accumulate across ingests |
| `sources` | UNION | Provenance accumulates |
| `concepts` | UNION | Related concept links accumulate |
| `claims` | APPEND — new claims added to list | Claims from each source are additive |
| `confidence` | SET if provided | LLM's assessment of the source |
| `last_updated` | SET to today | Always updated on enrichment |
| `title` | PRESERVE | Page identity not changed by enrichment |
| `summary` | PRESERVE | Page scope not changed by enrichment |
| `tldr` | PRESERVE | Page summary not changed by enrichment |
| `status` | PRESERVE | May have been set by human or lint |
| `contradictions` | PRESERVE | Wiki-managed relationship field |
| `type` | PRESERVE | Category not changed by enrichment |

**Why union for tags/read_when/sources?**
These fields accumulate knowledge from multiple ingests. Removing a value
added by a prior ingest would silently break search and context retrieval.
The LLM enriching from one source does not know what other sources added.
Append-only is safe; shrinking is not.

## Query Result — Body + Frontmatter

When `wiki ingest --analysis enrichment.json` processes a `QueryResult` entry,
a new page is created. This is the one case where the LLM writes a page body.

Frontmatter is generated from `QueryResult` fields:

| `PageFrontmatter` field | Source |
|-------------------------|--------|
| `title` | `query_result.title` |
| `summary` | `query_result.tldr` |
| `tldr` | `query_result.tldr` |
| `read_when` | `query_result.read_when` |
| `type` | `query-result` (always) |
| `tags` | `query_result.tags` |
| `sources` | `query_result.sources` |
| `status` | `active` (always, at creation) |
| `last_updated` | Today's ISO 8601 date |
| `confidence` | `medium` (default) |
| `contradictions` | `[]` |
| `claims` | `[]` |

The body is written exactly as provided by the LLM — no reformatting.
Query result slugs must start with `queries/`. Fails if slug already exists
(unless `--update` flag is set).

## Parsing an Existing File

`integrate` splits an existing `.md` file into frontmatter and body before
any enrichment or update. The format is always:

```
^---\n
<yaml block>
---\n
\n?
<body>
```

Parse rules:
- First line must be `---`
- Frontmatter ends at the next `---` line
- Everything after the closing `---` (including the blank line) is the body
- No frontmatter block → error for enrichment (enrichment requires an existing
  page); for direct ingest with `--update`, generate minimal frontmatter

---

## File Encoding and Line Endings

All wiki pages are UTF-8, LF line endings. `integrate` normalises CRLF to LF
on write and rejects non-UTF-8 body content with a validation error.
